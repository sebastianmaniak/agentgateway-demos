The Problem

Why Load Balance LLM Providers?

Single-provider architectures are fragile, expensive, and inflexible.

⚠️

Rate Limits

Hit your provider's TPM/RPM cap and your entire application stalls. No fallback. No recovery.

💸

Vendor Lock-in

Hardcoded to one provider? You can't leverage pricing differences, new models, or negotiate better terms.

🔍

No Comparison

How do you A/B test models? Without traffic splitting, you're guessing which model performs best.

02
Core Algorithm

Power of Two Choices

Not round-robin. Not random. Intelligent, adaptive routing.

Pick two random providers from the available pool for each incoming request
Score each provider using health, latency, and pending request count
Route to the winner — the provider with the higher score gets the request
Evict on 429s — rate-limited providers are temporarily removed from the pool
SCORING
// P2C scoring formula score = health / (1 + latency_penalty) // where latency_penalty = request_latency * (1 + pending * 0.1) // Health: EWMA success rate (α = 0.3) // Latency: EWMA of response time // Pending: active in-flight requests
Higher health + lower latency + fewer pending = higher score
03
Observability

How agentgateway Tracks Health

Three real-time metrics power every routing decision.

❤️

Health Score (EWMA)

Exponentially-weighted moving average of success rate. Each success = 1.0, failure = 0.0. Recent results weigh more heavily.

α = 0.3

⏱️

Request Latency (EWMA)

Average response time in seconds. Only successful requests contribute — failures are excluded to avoid skewing.

lower = better

📨

Pending Requests

Each active in-flight request adds a 10% penalty to the latency component. Naturally prevents overloading one provider.

+10% per request

Configuration

The Backend Resource

One YAML. Two providers. Automatic load balancing.

YAML
apiVersion: agentgateway.dev/v1alpha1 kind: AgentgatewayBackend metadata: name: loadbalanced-backend namespace: agentgateway-system spec: ai: groups: # providers in same group = P2C - providers: - name: openai-gpt4 openai: model: gpt-4o policies: auth: secretRef: name: openai-secret - name: anthropic-claude anthropic: model: claude-sonnet-4-6 policies: auth: secretRef: name: anthropic-secret
06
Routing

Gateway API HTTPRoute

Standard Kubernetes Gateway API — nothing proprietary.

YAML
apiVersion: gateway.networking.k8s.io/v1 kind: HTTPRoute metadata: name: loadbalanced-route spec: parentRefs: - name: agentgateway-proxy rules: - matches: - path: type: PathPrefix value: /chat backendRefs: - name: loadbalanced-backend group: agentgateway.dev kind: AgentgatewayBackend
Path-based routing — /chat maps to the load balanced backend
Standard Gateway API — works with any Gateway API compatible tooling
Custom backend type — references AgentgatewayBackend instead of a Service
Your app sends requests to /chat — agentgateway handles provider selection

Two separate backends — each with its own model configuration

Gradual rollout — shift weights as confidence grows: 80/20 → 50/50 → 0/100
YAML
rules: - matches: - path: value: /test backendRefs: - name: stable-backend group: agentgateway.dev kind: AgentgatewayBackend weight: 80 # ← 80% traffic - name: canary-backend group: agentgateway.dev kind: AgentgatewayBackend weight: 20 # ← 20% traffic
Good to Know

Eviction & Limitations

Understanding what triggers failover — and what doesn't.

Triggers Eviction

ResponseAction
429 Too Many Requests Provider evicted from pool
Rate-limit headers present Respects retry-after

Does NOT Trigger Eviction

ResponseAction
503 Service Unavailable Health score degrades only
Connection errors / DNS failures Health score degrades only
Timeouts Health score degrades only
Providers with degraded health still receive traffic — just less of it. Only 429s with proper rate-limit headers cause full eviction.

Takeaways

Four Things to Remember

1

P2C is smarter than round-robin

Adaptive scoring considers health, latency, and load. Not static. Not random.

2

Multi-provider is one YAML away

Put providers in the same group — agentgateway handles the rest. No SDK changes.

3

A/B testing uses standard Gateway API

Weighted backendRefs — no custom annotations, no proprietary config.

4

Observability is built in

OpenTelemetry metrics, health scores, and per-provider traces. Verify balancing is working.
